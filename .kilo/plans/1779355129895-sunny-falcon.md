# Plan de correction - Alignement des contrats de données Frontend/Backend

## Objectif
Assurer la conformité totale des contrats de données entre le backend ElysiaJS et le frontend React Router 7 via le SDK Hey API généré à partir de l'OpenAPI.

---

## Problèmes identifiés

### 1. Incohérence de naming (camelCase vs snake_case)

| Module | Backend retourne | SDK attend | Statut |
|--------|------------------|------------|--------|
| **Projects** | camelCase (`settingsJson`, `rootPath`) | snake_case (`settings_json`, `root_path`) | ❌ Non aligné |
| **Characters** | camelCase (`projectId`, `narrativeRole`) | snake_case (`project_id`, `narrative_role`) | ❌ Non aligné |
| **Story** | snake_case (`project_id`, `tome_id`) | snake_case | ✅ Aligné |
| **Warnings** | camelCase (`projectId`, `entityKind`) | snake_case (`project_id`, `entity_kind`) | ❌ Non aligné |
| **Versions** | camelCase (`projectId`, `branchName`) | snake_case | ❌ Non aligné |
| **Exports** | camelCase (`projectId`, `artifactPath`) | snake_case | ❌ Non aligné |
| **Generation** | À vérifier | snake_case | ⚠️ À vérifier |
| **Assets** | À vérifier | snake_case | ⚠️ À vérifier |

### 2. Structure des réponses listes

| Endpoint | Backend retourne | SDK attend | Statut |
|----------|------------------|------------|--------|
| `GET /projects` | `Project[]` (array direct) | `{ items: Project[] }` | ⚠️ Frontend gère les deux |
| `GET /projects/{id}/characters` | `Character[]` | `{ items: Character[] }` | ⚠️ À vérifier |
| `GET /projects/{id}/warnings` | `Warning[]` | `Warning[]` | ⚠️ À vérifier |
| `GET /projects/{id}/versions` | `Version[]` | `Version[]` | ⚠️ À vérifier |
| `GET /projects/{id}/exports` | `Export[]` | `Export[]` | ⚠️ À vérifier |
| `GET /projects/{id}/story` | `StorySummary` | `StorySummary` | ✅ Probablement OK |

### 3. Clés JSON dans les champs Json

Les champs `settingsJson`, `metadataJson`, `tagsJson`, etc. sont stockés en JSON dans PostgreSQL. Le frontend attend des clés en snake_case pour ces objets également.

---

## Principe de correction

**Règle d'or** : Le SDK Hey API génère automatiquement des types en snake_case car c'est la convention OpenAPI/JSON. Le backend DOIT retourner du snake_case pour être compatible.

Deux options s'offrent à nous :

### Option A : Modifier le backend pour retourner du snake_case (Recommandée)
Mettre à jour tous les `serialize*()` fonctions pour retourner du snake_case. C'est plus maintenable car :
- Le SDK est régénéré automatiquement
- Un seul endroit à modifier par module
- Cohérent avec les conventions API REST

### Option B : Configurer Hey API pour générer du camelCase
Nécessite une configuration spéciale du générateur et une régénération complète. Moins flexible pour l'écosystème.

**Décision** : Option A - Modifier le backend.

### Phase 4 : Correction des Response Wrappers (Listes)

**Analyse nécessaire** : Vérifier si le frontend attend des wrappers `{ items: [...] }` ou des arrays directs.

**Démarche** :
1. Analyser tous les endpoints de listes dans le backend
2. Comparer avec les types générés dans `types.gen.ts`
3. Standardiser :
   - Soit wrapper systématique `{ items: [...], total?: number }`
   - Soit arrays directs avec conventions de nommage

**Recommandation** : Utiliser des wrappers `{ items: [...] }` pour permettre l'ajout futur de métadonnées (pagination, total, etc.).

**Endpoints à corriger** :
- `GET /api/projects` → actuellement array, wrapper souhaitable
- `GET /api/projects/{id}/characters` → à vérifier
- `GET /api/projects/{id}/warnings` → à vérifier
- etc.

```json
{
  "id": "...",
  "name": "...",

## Questions ouvertes

1. **Faut-il wrapper toutes les listes avec `{ items: [...] }` ?**
   - Recommandé : Oui, pour cohérence et extensibilité
   - Impact : Nécessite modification backend + frontend

2. **Les champs JSON internes (settings_json) doivent-ils aussi être en snake_case ?**
   - Recommandé : Oui, pour cohérence totale
   - Impact : Migration potentielle des données existantes

3. **Faut-il créer un helper de sérialisation générique ?**
   - Option : Créer `snakeCaseKeys(obj)` dans `@lib/utils`
   - Avantage : Évite les erreurs manuelles
   - Inconvénient : Moins explicite, potentielles perfs
