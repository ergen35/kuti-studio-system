# Plan de Migration : De api.ts manuel vers SDK OpenAPI généré

## Contexte

Actuellement, le frontend utilise deux sources d'accès API :
1. **`app/lib/api.ts`** : API manuelle avec types et fonctions custom
2. **`app/lib/backend/@tanstack/react-query.gen.ts`** : SDK auto-généré par @hey-api/openapi-ts

L'objectif est de migrer complètement vers le SDK généré pour éviter la duplication et garantir la cohérence avec le backend.

---

## Analyse de l'état actuel

### Fichiers utilisant `api.ts`
- **Routes (40+ occurrences)** : `home.tsx`, `project.tsx`, `characters.tsx`, `character.tsx`, `story.tsx`, `scene.tsx`, `chapter.tsx`, `tome.tsx`, `assets.tsx`, `warnings.tsx`, `versions.tsx`, `exports.tsx`, `generation.tsx`, `settings.tsx`
- **Composants** : `ProjectCard.tsx`, `CharacterCard.tsx`, `CharacterImageGenerator.tsx`, `GenerationPanelGrid.tsx`, `SceneGenerationModal.tsx`, `SceneMangaGallery.tsx`, etc.

### Ce qui est importé depuis `api.ts`
| Élément | Usage | Migration vers |
|---------|-------|----------------|
| `API_BASE_URL` | URLs d'images/fichiers | `client.getConfig().baseUrl` ou garder export simple |
| `apiErrorMessage()` | Gestion erreurs | Créer helper utilitaire (`lib/errors.ts`) |
| `csv()` | Parsing CSV simple | Créer helper utilitaire (`lib/utils.ts`) |
| `ApiError` class | Type erreur | Utiliser erreurs du SDK ou créer wrapper |
| Types (`Project`, `Character`, `Tome`, etc.) | Typage props/states | Types générés (`types.gen.ts`) |
| `api` object | Appels API directs | Fonctions du SDK (`sdk.gen.ts`) ou hooks TanStack |

### Phase 1: Préparation - Créer les utilitaires remplaçants

#### Fichier: `app/lib/errors.ts`
```typescript
// Extraire apiErrorMessage du SDK
import { client } from "~/lib/backend/client.gen";

export function apiErrorMessage(error: unknown): string {
  if (error instanceof Error) {
    // SDK errors have response data
    const sdkError = error as { response?: { data?: { detail?: string } } };
    if (sdkError.response?.data?.detail) {
      return sdkError.response.data.detail;
    }
    return error.message;
  }
  return "Unexpected error";
}

export const API_BASE_URL = ((import.meta.env.VITE_KUTI_API_URL as string | undefined) || "http://127.0.0.1:8000").replace(/\/$/, "");
```

### Phase 3: Migration des appels API directs

#### Pattern API manuel actuel:
```typescript
// app/lib/api.ts
export const api = {
  projects: () => request<ProjectListResponse>("/projects"),
  project: (projectId: string) => request<Project>(`/projects/${projectId}`),
  createProject: (body: {...}) => request<Project>("/projects", { method: "POST", body }),
  // ...
};

// Usage dans composant
import { api } from "~/lib/api";
const data = await api.projects();
```

#### Pattern SDK TanStack Query:
```typescript
// Usage dans composant
import { useQuery, useMutation } from "@tanstack/react-query";
import { listProjectsOptions, createProjectMutation } from "~/lib/backend/@tanstack/react-query.gen";

// Query
const { data } = useQuery(listProjectsOptions());

// Mutation
const mutation = useMutation(createProjectMutation());
await mutation.mutateAsync({ body: { name: "..." } });
```

#### Pattern SDK Direct (si besoin hors React Query):
```typescript
import { listProjects, createProject } from "~/lib/backend/sdk.gen";

const { data } = await listProjects();
const { data } = await createProject({ body: { name: "..." } });
```

#### Conversions nécessaires:

| API Manuel | SDK TanStack Query | SDK Direct |
|------------|-------------------|------------|
| `api.projects()` | `listProjectsOptions()` | `listProjects()` |
| `api.project(id)` | `getProjectOptions({ path: { project_id: id } })` | `getProject({ path: { project_id: id } })` |
| `api.createProject(body)` | `createProjectMutation()` puis `mutate({ body })` | `createProject({ body })` |
| `api.characters(pid)` | `listCharactersOptions({ path: { project_id: pid } })` | `listCharacters({ path: { project_id: pid } })` |
| `api.character(pid, cid)` | `getCharacterOptions({ path: {...} })` | `getCharacter({ path: {...} })` |
| `api.story(pid)` | `getStorySummaryOptions({ path: {...} })` | `getStorySummary({ path: {...} })` |
| `api.createTome(pid, body)` | `createTomeMutation()` puis `mutate({ path, body })` | `createTome({ path, body })` |
| `api.createScene(pid, body)` | `createSceneMutation()` puis `mutate({ path, body })` | `createScene({ path, body })` |
| `api.assets(pid)` | `listAssetsOptions({ path: {...} })` | `listAssets({ path: {...} })` |
| `api.warnings(pid)` | `listWarningsOptions({ path: {...} })` | `listWarnings({ path: {...} })` |
| `api.versions(pid)` | `listVersionsOptions({ path: {...} })` | `listVersions({ path: {...} })` |
| `api.exports(pid)` | `listExportsOptions({ path: {...} })` | `listExports({ path: {...} })` |
| `api.generationJobs(pid)` | `listGenerationJobsOptions({ path: {...} })` | - |
| etc. | ... | ... |

**Points d'attention:**
- Les paths params passent de `projectId` à `project_id` (snake_case)
- Les query params de `tomeId` à `tome_id` (snake_case)
- Les mutations nécessitent un appel à `mutate()` ou `mutateAsync()`
- L'invalidation du cache se fait via les query keys du SDK

### Phase 4: Mise à jour des hooks existants OU suppression

#### Option A: Mettre à jour `use-api.ts` pour wrapper le SDK
Remplacer les implémentations actuelles par des wrappers autour du SDK:

```typescript
// app/hooks/use-api.ts (simplifié)
export function useProject(projectId: string) {
  return useQuery({
    ...getProjectOptions({ path: { project_id: projectId } }),
    enabled: !!projectId,
  });
}

export function useCreateProject({ onSuccess }: { onSuccess?: () => void } = {}) {
  const queryClient = useQueryClient();
  return useMutation({
    ...createProjectMutation(),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["listProjects"] });
      onSuccess?.();
    },
  });
}
```

**Avantages:**
- Migration transparente pour les composants
- Peut ajouter logique custom (invalidation cache, transformations)

#### Option B: Supprimer `use-api.ts` et utiliser SDK directement
Les composants importent directement depuis le SDK:

```typescript
// Dans chaque composant
import { getProjectOptions, createProjectMutation } from "~/lib/backend/@tanstack/react-query.gen";
import { useQuery, useMutation } from "@tanstack/react-query";
```

**Avantages:**
- Moins de couche d'indirection
- Types toujours à jour avec OpenAPI
- Pas de maintenance de wrappers

**Désicion utilisateur:** Préférez-vous Option A (garder des hooks wrappers) ou Option B (SDK direct dans composants) ?

## Questions essentielles avant démarrage

### 1. Gestion des erreurs
**Proposition:** Créer un helper `apiErrorMessage()` dans `lib/errors.ts` qui adapte les erreurs du SDK au format attendu par les composants.

### 2. Wrapper hooks vs SDK direct
**Option A:** Garder `use-api.ts` comme wrapper (migration transparente)
**Option B:** Utiliser SDK direct dans chaque composant (plus direct, moins de code)

### 3. Migration en une fois ou progressive
**Option A:** Tout migrer d'un coup (plus rapide mais risqué)
**Option B:** Par module (characters → story → assets → etc.)

### 4. Types
Les types du SDK (`types.gen.ts`) peuvent différer légèrement des types manuels. **Besoin d'une revue type par type ?**
